# GPT-5.6 Prompt Guidance 摘要与落地检查表

> 来源：[Prompting guidance for GPT-5.6](https://developers.openai.com/api/docs/guides/prompt-guidance-gpt-5p6)
>
> 核对日期：2026-07-15
>
> 文档性质：中文提炼与工程化解读，不是原文镜像或逐字翻译。

## 适用范围

这篇指南面向 GPT-5.6，但其中关于提示词结构、工具路由、长流程状态、验收和迭代的方法，也适用于 Claude Code Agent、Skill 和 Hook 的设计。

跨模型使用时必须区分两类内容：

- **可迁移原则**：减少重复、结果优先、明确成功标准、工具边界、停止条件、证据要求和验证闭环。
- **OpenAI API 特有能力**：`text.verbosity`、GPT-5.6 reasoning effort、Programmatic Tool Calling、Responses API phase/state 等。不能把这些参数原样写进 Claude Code 配置。

## 一页结论

高质量提示词不是把所有可能步骤都写进去，而是用尽量短的契约回答七个问题：

1. 模型要交付什么结果？
2. 什么条件成立才算完成？
3. 哪些约束是真正不可违反的？
4. 哪些工具在什么条件下使用？
5. 哪些证据必须先取得？
6. 失败时何时重试、降级、询问或停止？
7. 完成前如何验证？

对于已经能工作的提示词，先建立基线，再一次只删除或修改一组指令。不要同时换模型、重写提示词、调整工具集和改变推理强度，否则无法定位回归来源。

OpenAI 在内部 coding eval 中观察到的方向值约为：评分提高 10%-15%，总 token 降低 41%-66%，成本降低 33%-67%。这些结果不是通用保证，也不应直接当作本项目目标；任何删减都必须在 Anban 的代表性任务上重新评估。

## 核心原则

### 1. 先做减法

优先删除：

- 同一规则的重复表述；
- 不改变行为的风格或过程说明；
- 不改变行为的示例；
- 模型已经稳定做到的通用步骤；
- 与当前任务无关的工具和工具描述。

必须保留：

- 用户可见的最终结果；
- 成功标准和停止条件；
- 安全、业务、证据、权限约束；
- 依赖上下文判断的工具路由规则；
- 必需输出结构和验证要求。

删除后要检查剩余规则是否互相矛盾。对强指令遵循模型而言，冲突规则通常比缺少细节更不稳定。

### 2. 描述目的地，不要穷举路线

优先写“需要达到什么状态”，把可变路径交给模型判断。例如：

```text
目标：完成文章草稿并登记所有交付物。

成功标准：
- 正文、图片清单、质量报告和 draft.json 均已落盘；
- 所有必需证据和质量门槛都有可读取的记录；
- 允许的重试已经完成；
- 阻塞项包含失败阶段、已保留产物和恢复入口。
```

只有安全红线、必需字段和绝不能发生的动作适合使用“必须”“禁止”“仅允许”。搜索、重试、询问、工具选择等判断型行为更适合使用决策规则。

### 3. 显式写停止条件

长流程需要明确区分：

- 结果已足够，可以停止；
- 缺少一个最小事实，需要补充检索；
- 可重试错误，仍在预算内；
- 可降级错误，降级不会破坏核心结果；
- 硬阻塞，必须保留现场并停止；
- 需要外部授权或扩大范围，必须请求确认。

不要把“减少工具循环”放在正确性、必需证据、计算、引用或验收之前。

### 4. 分开人格与协作方式

- **人格**控制语气、温度、直接程度、正式度和润色风格。
- **协作方式**控制何时询问、何时假设、主动执行到什么程度、如何解释权衡、如何验证和处理不确定性。

两者都应简短，不能替代目标、工具规则和成功标准。与其写“友好”“有同理心”，不如写可观察的表达选择，例如“先直接给出结论；用户报告故障时先复述具体问题；不使用泛化赞美”。

### 5. 定义自治与授权边界

推荐按请求类型定义权限：

- 回答、解释、评审、诊断、规划：读取和分析相关材料，但不自动实施修改。
- 修改、构建、修复：完成范围内的本地更改和非破坏性验证，无需反复询问。
- 外部写入、破坏性动作、付费动作、权限变化或显著扩大范围：执行前确认。

规则集中写一次。重复“先询问”“不要修改”容易让模型在安全的预期动作上也频繁暂停。

### 6. 精简并约束工具路由

只暴露当前任务需要的工具。工具说明至少覆盖：

- 工具做什么；
- 什么时候调用；
- 关键返回字段；
- 空结果、部分结果和错误如何处理。

依赖检索的动作必须先完成发现、读取和验证。独立读取可以并行；后续动作依赖前一步结果时保持顺序执行；并行结果返回后先综合，再执行写操作。

空结果不等于事实不存在。结果为空、异常窄或明显不完整时，允许一到两个有意义的替代检索，再决定停止。

### 7. 只在有界的确定性归约中使用程序化调用

Programmatic Tool Calling 适合由代码处理大量或多批结构化结果，再只返回紧凑 schema 的阶段，例如过滤、连接、排序、去重、聚合、批处理和重复的确定性校验。

以下场景优先直接工具调用：

- 一次调用已经足够；
- 中间结果很小；
- 每个结果都可能改变下一步决策；
- 动作需要授权；
- 最终结果必须保留原生 artifact 或引用；
- 调用之间需要语义判断。

不能只写“高效使用程序化工具调用”。必须限定可使用的工具、处理阶段、输出 schema、重试次数、停止条件，以及何时把控制权交回模型判断。在 Claude Code 中，这一原则更适合映射为有界 helper script 或 command Hook，而不是让 Agent 用临时脚本绕过 MCP。

### 8. 给证据和检索设置预算

需要事实依据的任务，应定义：

- 哪些结论必须有来源；
- 什么证据算充分；
- 引用应该贴在哪个结论后；
- 多个来源冲突时如何报告；
- 缺少证据时缩小结论还是停止。

普通问题先做一次短关键词检索。只有缺少必需事实、用户要求穷举、必须读取指定文件或关键结论尚无依据时，才继续检索。

创作任务也要区分事实和创意表达。不能为了让内容更有说服力而虚构名称、指标、日期、客户结果或产品能力。

### 9. 长流程使用稀疏、结果型进度

首次工具调用前给一到两句简短说明。后续只在主要阶段变化或新发现改变计划时更新，并同时说明：

- 已完成的具体结果；
- 下一步动作。

不要逐条复述常规工具调用。状态压缩应发生在主要里程碑之后，而不是每一轮都压缩；持久化的旧推理只在目标、假设和优先级仍稳定时复用。可缓存的大段前缀应尽量稳定，避免无意义地改写固定系统提示词。

### 10. 先补契约，再提高推理强度

提高 reasoning effort 前，先检查提示词是否缺少：

- 成功标准；
- 依赖或前置条件；
- 工具路由；
- 验证循环。

推理强度应通过代表性任务评估。高强度不是全局默认值，只在质量收益能够覆盖延迟和成本时使用。

### 11. 前端与视觉任务保留产品约束

更强的视觉判断不等于可以忽略现有系统。增量视觉任务仍应：

- 先检查现有 design token、组件和模式；
- 不增加用户未要求的功能或装饰；
- 保持响应式行为和必需状态；
- 渲染并实际检查结果。

视觉理解、OCR 或坐标敏感任务应根据图像密度和空间精度选择合适输入质量。额外成本只有在确实提高验收可靠性时才合理。

### 12. 完成前必须验证

代码任务通常验证：

- 受影响行为的针对性测试；
- 类型检查或 lint；
- 受影响包的构建；
- 完整验证过重时的最小 smoke test。

视觉和文档产物需要先渲染，再检查裁切、间距、缺失内容、可读性和一致性。无法运行验证时，必须说明原因和替代检查。

## 推荐提示词结构

复杂 Agent 或 Skill 可以从以下结构开始，每节只保留会改变行为的内容：

```text
Role: 模型的职责和业务上下文
Personality: 语气与协作方式
Goal: 用户可见结果
Success criteria: 完成前必须成立的条件
Constraints: 安全、业务、证据和副作用边界
Tools: 使用条件、禁止路径、关键返回和错误语义
Output: 文件、字段、格式、长度和交付方式
Stop rules: 重试、降级、询问、保留现场和停止条件
```

在 Claude Code 中不必机械保留英文标题，但应确保这七类信息各自只有一个权威位置。

## Agent、Skill 与 Hook 的落地映射

| 表面 | 应保留 | 应移出 |
|---|---|---|
| Agent description | 负责的任务、明确触发边界、排除项 | 完整流程、长关键词列表、质量细节 |
| Agent body | 目标、阶段边界、路由、成功和停止条件 | 专业方法论、重复 schema、长示例 |
| Skill description | 做什么、何时用、何时不用 | 营销式介绍、重复同义词、全流程说明 |
| `SKILL.md` | 核心决策规则、输入输出、引用导航 | 长 API 文档、示例库、完整 rubric |
| supporting files | 按需读取的详细方法、schema、示例 | 每次都必须加载的核心契约 |
| command Hook | 确定性检查、格式化、权限拦截、文件存在性 | 需要复杂语义判断的开放任务 |
| prompt Hook | 基于现有 hook 输入即可判断的单次语义决策 | 需要读文件、跑命令或多轮修复的任务 |

## 迁移工作流

1. 固定当前模型、推理配置和工具集，建立代表性任务基线。
2. 记录成功率、产物完整率、工具调用数、turn、token、成本、耗时和重试次数。
3. 删除一组重复或无效指令。
4. 用相同任务重新评估。
5. 只有出现可复现回归时，增加最小的针对性规则。
6. 单独比较修改前后版本，确认质量提升不是由更多 token 或更多重试换来的。
7. 最后再调整模型、推理强度或工具集。

## Prompt Review 检查表

- [ ] 目标能用一句话说明。
- [ ] 成功标准是可观察、可验证的状态。
- [ ] 每条绝对规则都是真正不变量。
- [ ] 没有同一规则的多次改写。
- [ ] 没有互相冲突的停止、降级或询问规则。
- [ ] 工具调用有明确前置条件和错误语义。
- [ ] 事实结论有证据要求，缺失时不会猜测。
- [ ] 输出格式只有一个权威 schema。
- [ ] 长流程只有阶段级进度，不逐工具旁白。
- [ ] 验证步骤和无法验证时的处理已定义。
- [ ] 变更能通过同一组代表性任务与旧版本比较。
